🌊 User Profile Page (profile.html)

Page Overview

The User Profile page is an interactive JavaScript-driven profile builder that collects user information through progressive prompts and displays it dynamically. This page demonstrates DOM manipulation, event handling, and state management.

HTML Structure

Key HTML tags used and their purpose:

  • <header> & <nav> - Shared template navigation with active page indication
  • <main> - Contains all profile-building interface elements
  • <div class="step-card"> - Interactive cards for each profile step
  • <div class="progress-bar"> - Visual completion indicator
  • <div class="profile-output"> - Container for dynamically displayed user data
  • <button> - Control buttons for step navigation and returning to skipped steps
  • <footer> - Contains links to Page Editor and Validation pages

CSS Techniques (Embedded - 5+ Rules)

/* Rule 1: Step card styling with gradient background */ .step-card { background: linear-gradient(135deg, #006994 0%, #00a1d9 100%); transition: transform 0.3s ease; } /* Rule 2: Hover effect with elevation */ .step-card:hover { transform: translateY(-5px); box-shadow: 0 8px 25px rgba(0,105,148,0.4); } /* Rule 3: Progress bar animation */ .progress-fill { animation: progressPulse 2s infinite; } /* Rule 4: Profile output container styling */ .profile-output { border: 3px solid #006994; box-shadow: 0 10px 40px rgba(0,105,148,0.2); } /* Rule 5: Button styling with hover effects */ .control-btn { transition: all 0.3s ease; text-transform: uppercase; letter-spacing: 1px; }

JavaScript Implementation

The page uses JavaScript for:

  • Prompt-based data collection: Three steps with four prompts each (12 total prompts)
  • State management: profileData object tracks all user inputs
  • Progress tracking: Dynamic progress bar updates based on completed steps
  • Skip functionality: Users can skip steps, marked visually and tracked
  • Return to skipped steps: Option to complete previously skipped information
  • DOM manipulation: Profile sections reveal progressively as data is added

🗺️ Sitemap Page (sitemap.html)

Page Overview

The Sitemap page uses SVG (Scalable Vector Graphics) to create a visual representation of the website structure. Each node is clickable and links to the corresponding page, demonstrating both technical SVG implementation and interactive navigation.

HTML Structure

  • <svg> - Main container with viewBox for responsive scaling
  • <a> wrapped around <g> - Creates clickable nodes that open in new tabs
  • <rect> and <circle> - Different shapes distinguish page types
  • <text> - Labels for each node
  • <line> - Connects nodes to show hierarchy
  • <title> - Provides tooltips and screen reader text

SVG Implementation Details

  • Hierarchy: Splash → Home → All other pages with clear parent-child relationships
  • Node types: Rectangles for main pages, circles for content pages, small rectangles for editors
  • Color coding: Dark blue (main), light blue (content), green (Student 4 pages), gray (utility)
  • Interactive features: Hover effects change fill color and add gold border; focus states for keyboard navigation
  • Accessibility: tabindex="0" on each node group, ARIA labels, and title elements

CSS for SVG

/* Hover effect for all nodes */ .node:hover rect, .node:hover circle { fill: #00a1d9; stroke: #ffd700; stroke-width: 3; filter: drop-shadow(0 4px 8px rgba(0,0,0,0.2)); } /* Focus state for keyboard navigation */ .node:focus rect, .node:focus circle { stroke: #ffd700; stroke-width: 4; }

Responsiveness

The SVG uses viewBox="0 0 900 700" and width:100% to scale proportionally across devices. The container has overflow:visible to prevent clipping on smaller screens.

💻 Technical Implementation Summary

HTML5 Semantic Structure

All my pages use semantic HTML5 elements including header, nav, main, section, and footer. This improves readability, SEO, and accessibility.

CSS Techniques

  • Flexbox & Grid: Used for responsive layouts (step cards, image grids, profile sections)
  • Gradient backgrounds: Applied to cards for visual appeal
  • Transitions & transforms: Smooth hover effects on interactive elements
  • Keyframe animations: Progress bar pulse animation
  • Position sticky/fixed: Internal navigation and "Go to Top" button
  • Media queries: Responsive design for mobile devices

JavaScript Features

  • Object data structure: Profile data stored in nested objects
  • Functions: Modular functions for each step and updates
  • Conditional logic: Handling skipped vs completed steps
  • DOM manipulation: Updating displayed content and progress bar
  • Event handling: Button clicks trigger prompt sequences

🔧 Challenges and Lessons Learned

Challenge 1: SVG Accessibility

Problem: Making SVG elements keyboard-accessible and screen-reader friendly was initially difficult. Simply adding links inside SVG didn't provide proper focus management.

Solution: Added tabindex="0" to each group, included title elements, and used ARIA labels. Tested with keyboard navigation to ensure all nodes are reachable.

Lesson: SVGs need explicit accessibility considerations; they don't inherit accessibility features automatically.

Challenge 2: State Management in Profile Builder

Problem: Tracking which steps were completed vs skipped while allowing users to return to skipped steps required careful state design.

Solution: Created separate objects for profileData, completedSteps, and skippedSteps. Added functions to update all three consistently.

Lesson: Planning data structures before coding saves time and reduces bugs.

Challenge 3: Responsive SVG

Problem: The sitemap SVG was too large on mobile and text became unreadable.

Solution: Used viewBox instead of fixed width/height, set SVG width to 100%, and tested on multiple screen sizes. Considered but decided against separate mobile version.

Lesson: SVGs with viewBox scale beautifully if designed with responsiveness in mind from the start.

Challenge 4: User Experience with Prompts

Problem: Multiple consecutive prompts could frustrate users.

Solution: Added clear step indicators in prompt messages, allowed skipping, and provided positive feedback after each step completion.

Lesson: Even with technical requirements, user experience matters. Small touches like confirmation messages improve engagement.

♿ Accessibility Considerations

Semantic HTML

  • Proper heading hierarchy (h1 → h2 → h3) for screen reader navigation
  • Landmark elements (header, nav, main, footer) for easy page navigation
  • Lists (ul, ol) used where appropriate for structured content

Images

  • All images include descriptive alt text (e.g., "Fisherman using traditional sustainable fishing methods")
  • Decorative images in profile avatar use empty alt or are handled via CSS

Keyboard Navigation

  • Sitemap nodes have tabindex="0" and visible focus states
  • All buttons and links are keyboard accessible
  • "Go to Top" button can be activated via keyboard
  • Logical tab order through all interactive elements

Color and Contrast

  • High contrast between text and backgrounds (dark blue on light blue/white)
  • Color not used as sole indicator (icons and text also convey meaning)
  • Hover states provide additional visual feedback

Screen Reader Support

  • SVG nodes include title elements and ARIA labels
  • Form fields in feedback page (if applicable) have associated labels
  • Meaningful link text (not just "click here")

Responsive Design

  • All pages are usable at 200% zoom
  • Mobile layouts stack content appropriately
  • Touch targets are at least 44x44px on mobile

5. Academic Integrity

  • All code is original or properly attributed
  • References included for tutorials and resources consulted

  • Validation pages demonstrate commitment to web standards

By following these guidelines, my pages are suitable for publication on university servers and comply with JANET regulations.
